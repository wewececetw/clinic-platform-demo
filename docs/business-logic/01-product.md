# 01. 產品定位與角色

## 1.1 產品定位

**產品**：醫療院所門診流程管理平台（Multi-tenant SaaS）

### 解決問題
- 傳統診所候診體驗差（不知等多久、叫號混亂、抽號紙本化）
- 流程僵化：報到 → 看診 → 領藥順序寫死在系統裡，不同科別 / 診所流程差異無法客製
- 櫃台人員壓力大：重複性操作（叫號、過號、查詢）耗時
- 缺乏智慧決策輔助：分流靠經驗、等候時間靠猜

### 差異化價值

| 價值主張 | 實作機制 |
|---------|---------|
| **流程可客製** | Workflow Engine（有向圖）讓每家診所自訂步驟與跳轉條件 |
| **多租戶隔離** | 所有資料表含 `ClinicId`，資料與流程設定 per-clinic |
| **AI 輔助決策** | LLM 分流科別、自然語言下達指令、未來 RAG 輔助診斷 |
| **即時同步** | SignalR 推播候診狀態給病患 / 護理師 / 醫師 |
| **三種報到方式** | OTP 簡訊、QR Code、櫃台手動（涵蓋不同到院情境） |

### 目標使用情境
- 中小型診所（1–10 位醫師）：看診 + 領藥流程
- 連鎖診所：多院所共用平台，各院所獨立設定

---

## 1.2 使用者角色

| 角色 | Role Code | 說明 |
|------|-----------|------|
| 病患 | `Patient` | 到院看診者，可匿名（Manual 報到無電話時） |
| 護理師 / 櫃台 | `Nurse` | 管理候診、手動報到、過號 |
| 醫師 | `Doctor` | 呼叫下一位、看診、開處方 |
| 藥師 | `Pharmacist` | 領取處方、配藥、發藥 |
| 管理員 | `Admin` | 設定工作流、診所設定、檢視營運數據 |

> 使用者可擁有多個角色（User ↔ UserRole ↔ Role 多對多）。
> Patient 非系統登入帳號，以 `Patient` entity 為準（`IsAnonymous` 可為 true）。
